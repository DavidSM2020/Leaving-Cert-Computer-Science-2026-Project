Basic Requirements

  1. For the first basic requirement, the digital input used was a button on the physical embedded system, which turned it on, and the analogue input was the thermometer built into the micro:bit.
  2. The second basic requirement asked for data collection - this gets completed by a datalogger on the micro:bit and it gets stored to a localised CSV file
  3. With the third basic requirement, I used real-time calculations based on immediately collected data - after normalising the information, the micro:bit would display a symbol indicating whether the drought in the soil was probable.

Advanced Requirements

  1. For the first advanced requirement, I implemented serial communication between the micro:bit and a local Python script. The micro:bit writes temperature and moisture readings and CSV's over USB serial which the Python script reads in real-time, processing said information and logs it.
  2. With the second advanced requirement, I developed a Python data analysis script `main_graphing.py` which loads CSV data and cleans it, calculating summary statistics and providing a visualisation.
  3. I decided the keep my adaptive system quite simple as there was no need for it to be overcomplicated. For the adaptive system - I implemented a sound warning depending on the drought probability calculated based of normalised values from the temperature and moisture.

Investigation

When I started off making my project, my first thought was to work on light pollution as my theme. I was going to measure light intensity using an LDR or photodiode connected to a micro:bit since the supplies were readily available at my school and there is plentiful publicly available data surrounding this specific environmental topic. Unfortunately, after some brief research and further consideration, I concluded that light pollution wasn't sufficient, and I had to change my idea. I would have technically been able to create a project on light pollution, but it lacked enough depth to get the amount of marks I was looking for due to the general simplicity of light pollution restricting me from going beyond basic requirements and particularly the what-if simulations from the advanced requirements which would have been difficult to make meaningful light intensity data as it was.

I shifted my focus to drought and soil conditions, which was a provided example given and proved to be a much better option with further complexity. Since soil moisture and temperature are both simply measurable using a micro:bit, the correlation between them is well documented, and drought risk is a relatively universal real world problem with genuine consequences in environmental and agricultural areas.

With the embedded system, I researched existing soil moisture sensor setups for the micro:bit and found that a simple makeshift sensor using two screws connected via crocodile clips (noted in my references). The micro:bit reads this as an analogue value between 0 and 1023 on pin P0, with lower values being indicative of drier soil. The micro:bit's built-in thermometer was used for temperature, and it was convenient since it didn't require an external sensor.

I also considered implementing a rain gauge using a weight sensor to measure rainfall. This would have added another variable to the drought model, but after considering the intricacies of including another variable - I concluded that it would be unnecessary for me to implement as soil moisture and temperature were sufficient.

There already exist some commercial soil moisture and drought prediction system which make use of a combination of satellite data, ground sensors, and modeling to provide real-time analytics for agriculture and water management. There are even indexes like the Palmer Z-Index and Palmer Crop Moisture Stress Index (CMSI) which exist to measure drought on varying scales.

Plan and Design

My design primarily involves two components working together: the physical embedded system (micro:bit), and the Python computer models for data logging, analysis and visualisation. These communicate through USB serial at 115200 baud (the default for V2 micro:bits).

For the computer model, Python was my language of choice since it has a wide array of libraries for handling data and it was the language I am the most confident with it. Through the use of USB serial, I got a seamless data transfer system.

The sensors used are the micro:bit's built-in thermometer (reading temperature in C), a makeshift soil moisture sensor using screws and crocodile clips connected to pins P0 and P1, and the micro:bit's built-in buttons A and B as digital inputs for starting and stopping the system.

The system follows this logic:

[Press Button A] -> [Start loop]
       ↓
[Read temperature from built-in sensor]
[Pulse P1 HIGH -> Read moisture from P0 -> Pulse P1 LOW]
       ↓
[Normalise: moisture_factor = 1 - moisture / 1023]
[Normalise: temp_factor = temp / 40]
       ↓
[Calculate: drought_prob = 0.65 * moisture_factor + 0.35 * temp_factor]
       ↓
[Send temp + moisture over serial to Python]
       ↓
[Display on LED matrix + play sound based on risk level]
    High  (>0.7)  -> X icon    + 698Hz tone
    Medium (>0.5) -> Neutral icon + 440Hz tone
    Low   (≤0.5)  -> Tick icon + no sound
       ↓
[Python logs to CSV -> analyse -> visualise]
       ↓
[Press Button B] -> [Stop]

As the creator of this system, I think that there are a few careers/end users who could find something like this useful. My main thoughts would be for a farmer, gardener, and environmental researcher tracking drought patterns. Each of these users need a system that is simple to interpret at a glance, and to me this informed the decision to use clear visual indicators on the LED matrix (tick, neutral face, X) which we are all familiar with. The addition of audio alerts instead of just raw numbers also aids in making the output immediately understandable without needing to read the data manually. I imagine that there are a lot more detailed and thorough utilities available that would work better but considering my limited time frame, I do believe that I did a good job on this.

I used a couple of different technologies with them being: MicroPython running on the micro:bit V2 for the embedded system, Python 3 on a connected computer for the model, with the pandas library for handling all the data, matplotlib for metric visualisation, and numpy was used for numerical operations. Data is stored in a CSV format and is transferred over the two systems using USB serial at 115200 baud.

When considering the embedded system hardware, I chose the micro:bit over other alternatives like Arduino or a Raspberry Pi. For me and my project, I considered the micro:bit to be the most practical choice given school availiblity, the convenience of built-in sensors, and its native MicroPython support which made development straightforward. When comparing the micro:bit to its alternatives, it was by far the simplest and there was no need for me to add external components for equivalent functionality, and a Raspberry Pi was just excessive for an elementary task.

Create

Development started off with setting up the soil moisture sensor on the micro:bit in MakeCode, then converting the logic to MicroPython. The normalisation formula was developed to convert raw sensor values (0–1023) into more manageable 0–1 factors. A weighted drought probability formula was chosen which gave moisture 65% of the weight and temperature 35%, since soil moisture is a more impactful indicator of a soil sample's drought conditions.

Serial output was then added so Python could receive the data in real-time. main_reading.py was built to listen on the serial port and log to a CSV file, followed by main_graphing.py to analyse and visualise the collected data. Finally, main_whatif.py was written to simulate what-if scenarios without needing the hardware running.

Challenges encountered during development included:

  • The micro:bit's built-in datalogger conflicted with serial logging - both could not run simultaneously, so serial was chosen as it allowed real-time data transfer to Python. This is noted as a comment in the micro:bit code.
  • Normalising the moisture sensor required care, as the raw value ranges from roughly 50 (very dry) to 1023 (fully saturated), and needed to be inverted so that a higher output value represents greater drought risk.
  • Choosing the weighting for the drought probability formula required research into which environmental factors most contribute to soil drought conditions.

To develop further on the problem I encountered with my datalogger conflicting with serial logging, I should start by saying that it was the most significant issue and decision I had to deal with when finalising this project. The actual issue at hand was that the micro:bit was not capable of logging USB serial and physically datalogging at the same time. When both were active simultaneously, the datalogger intervened with the serial stream meaning that only the onboard datalogger was working and the Python model wasn't able to read any data off the system. After some troubleshooting, I identified this as the issue since it was initially quite unclear. I had to choose between the two and in the end, I opted for serial logging since it allowed real-time data analysis and was more influential in my project's success, deeming it to be more valuable. The commented out data logger code continues to remain in the micro:bit script to archive this decision.

Some executive decisions I made during the development of my embedded system include:
- Choosing serial over the datalogger
- Deciding against a rain gauge
- Keeping the weighting formula simple as opposed to introducing additional variables like humidity or sunlight

The core of the Python model is a weighted probability forumla drought_prob = 0.65 * moisture_factor + 0.35 * temp_factor. Both of the inputs (moisture and temperature) are first normalised on a 0 - 1 scale with moisture being inverted since higher raw sensor values were indicative of moisture soil. After the formula would be ran, it would return a result on a scale of 0 - 1 which are categorised into low, medium, and high risk. The what-if model furthers this by running the formula across all collected readings with modified inputs, which nicely allows for predictions to be made even without the hardware being currently present.

I kept my testing thorough but also simple. I used different methods, like a warm and cold hairdryer to emulate heat, and different soil from my garden to simulate drought! The results from my testing came out to be quite conclusive, and it matched my hypothesis. When testing with my hairdryer on warm settings,the temperature read was high and the micro:bit registered it as being at risk of drought, once I began using the cool setting on said hairdryer, the micro:bit said it wasn't at risk (displaying a tick). After testing with a hairdryer to simulate the two edges of temperature, I tested 3 times with different conditions.

The first condition was in normal soil and there was no risk of drought in the soil (it had rained the night prior), I also tested in a glass of water which told me that it was highly moist and maxed out the moisture sensor (1023) on the micro:bit. After testing with water, I also tested by holding it in the air, where, weirdly enough, it read to be quite in the middle with varying results but on average, approximately 600 (on a scale of 0 to 1023). Each test also responded as expected for the third advanced requirement (playing a sound based on conditions).

Through a mixture of testing and research mentioned in references, I also managed to get a relatively optimal formula for the weighting of each measured variable and condition (temperature and soil moisture)

Evaluation

If I had more time available to work on the project, I would have liked to implement a water gauge system to measure how absorbent soil is. Although, in the end, I am quite happy with the work I have accomplished with my project. I finished mostly everything that I wanted to get done but I might have been capable of doing more with more efficient time management.

My project successfully meets all three basic and the three advanced requirements were met sufficiently and completed. The embedded system collects real sensor data, and uses real-time processing to display it as a result. The Python model analyses collected data and runs what-if scenarios as well as an adaptive feedback mechanism playing sound alerts depending on if specific conditions are met.

Everything works well, the serial bus between the micro:bit and Python works reliably. The normalisation approach which I used is clean and the weighted formula produces sensible results deduced from testing and research. I implemented 3 what-if scenarios which genuinely demonstrate how the model behaves under varying circumstances. Thorough testing confirmed the system responded correctly across a range of real inputs.

There are a few limitations like how the moisture sensor is makeshift and can be inconsistent, two screws and crocodile clips are not weather proof or calibrated which means that reading can be inconsistent which leaves room for inaccuracy. The micro:bit's onboard thermometer can be slightly affected by the device's own running heat generation which means temperature readings might not be perfectly accurate (it is quite negligible tho). The model also only uses two inputs and variables which is a crude simplification of actual soil drought.

There is naturally room for improvement and the first future possible improvements which come to mind are a more external thermometer since the current one cannot be moved without moving the physical micro:bit. As stated already, I would have also liked to add a water gauge to measure rainfall. Some other things which would've been nice are: adding humidity as a third variable, and potentially connecting to an open source weather data to make the model's predictions more accurate over a period of time.

import time
import csv

# connect to microbit
ser = serial.Serial("/dev/ttyACM0", 115200)

filename = "drought_data.csv"

# create CSV header
with open(filename, "w", newline="") as f:
    writer = csv.writer(f)
    writer.writerow(["temp", "moisture", "moisture_factor", "temp_factor", "drought_prob"])

print("Started logging")

while True:

    line = ser.readline().decode("utf-8", errors="ignore").strip()

    if line == "":
        continue

    print("Raw:", line)   # shows exactly what the microbit sent

    try:
        parts = line.split(",")

        temp = float(parts[0])
        moisture = float(parts[1])

        # normalising
        moisture_factor = 1 - moisture / 1023
        temp_factor = temp / 40

        drought_prob = 0.65 * moisture_factor + 0.35 * temp_factor

        print(f"Temp: {temp} | Moisture: {moisture} | Risk: {drought_prob:.2f}")

        with open(filename, "a", newline="") as f:
            writer = csv.writer(f)
            writer.writerow([temp, moisture, moisture_factor, temp_factor, drought_prob])

    except Exception as e:
        print("Error:", e)

    time.sleep(1)

import pandas as pd
import matplotlib.pyplot as plt
import numpy as np

# Load real collected data

file_name = "drought_data.csv"
data = pd.read_csv(file_name)

data.columns = data.columns.str.strip()

cols = ["temp", "moisture", "moisture_factor", "temp_factor", "drought_prob"]
for col in cols:
    data[col] = pd.to_numeric(data[col], errors="coerce")

data = data.dropna()


def drought_prob_calc(temp, moisture):
    """Recalculate drought probability from raw sensor values."""
    moisture_factor = 1 - moisture / 1023
    temp_factor = temp / 40
    return 0.65 * moisture_factor + 0.35 * temp_factor


def risk_level(x):
    if x > 0.7:
        return "High"
    elif x > 0.5:
        return "Medium"
    else:
        return "Low"


# What-If Scenario 1: Temperature Increase
# Simulates a heatwave — what if temperatures rose by +5C and +10C?

scenario1 = data.copy()
scenario1["drought_prob_plus5"] = scenario1.apply(
    lambda row: drought_prob_calc(row["temp"] + 5, row["moisture"]), axis=1
)
scenario1["drought_prob_plus10"] = scenario1.apply(
    lambda row: drought_prob_calc(row["temp"] + 10, row["moisture"]), axis=1
)

# What-If Scenario 2: Critically Dry Soil
# Simulates a drought — what if soil moisture dropped to 200 or 50?

scenario2 = data.copy()
scenario2["drought_prob_dry"] = scenario2.apply(
    lambda row: drought_prob_calc(row["temp"], 200), axis=1
)
scenario2["drought_prob_very_dry"] = scenario2.apply(
    lambda row: drought_prob_calc(row["temp"], 50), axis=1
)

# What-If Scenario 3: Combined Worst vs Best Case
# What if both temp rose +8C AND soil was critically dry (moisture = 100)?

scenario3 = data.copy()
scenario3["drought_prob_worst"] = scenario3.apply(
    lambda row: drought_prob_calc(row["temp"] + 8, 100), axis=1
)
scenario3["drought_prob_best"] = scenario3.apply(
    lambda row: drought_prob_calc(row["temp"] - 3, 900), axis=1
)
